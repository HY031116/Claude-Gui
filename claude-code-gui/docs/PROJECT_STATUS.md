## 一、项目目标定义

### 核心定位
以 **Claude Code CLI** 为唯一内核，通过 Electron 桌面应用提供可视化界面。保留 Claude Code 的全部原生能力（工具调用、MCP、Agents、扩展思考等），同时在交互体验上对标 **OpenAI Codex CLI** 的极简对话流范式。

### 关键差异点

| 产品 | 界面哲学 | 核心优势 | 劣势 |
|:---|:---|:---|:---|
| **Claude Code CLI** | 纯终端交互 | 功能最全、更新最快、官方原生 | 无可视化、学习曲线陡峭 |
| **OpenAI Codex CLI** | 对话流即界面 | 极简体验、Turn 工作流、内嵌 Diff | 功能有限、非 Claude 生态 |
| **本项目目标** | **对话流即界面 + Claude 全能力** | 原生 Claude 能力 + Codex 式体验 | 需大量界面重构 |

## 三、技术总监评估

### 架构健康度

| 维度 | 评分 | 评价 |

| 进程架构 | ✅ 良好 | Electron 双进程 + IPC 通道设计合理 |

- **潜在风险**：列表型面板（MemSearch/History）数据量大时无虚拟滚动

- ✅ `nodeIntegration: false`, `contextIsolation: true`

### 5.2 五大核心差距

#### 差距 1：界面范式 —— 从「IDE 大全」到「对话即界面」

**Codex 哲学**：打开应用就是一个对话区域，所有操作以卡片形式嵌入对话流。无侧边栏导航，无独立面板切换。

**当前状态**：18 个左侧导航项，用户需不断跳转面板。聊天区域被挤在右侧。

**冲突本质**：Codex 的极简哲学 vs 当前「功能大全」哲学。

#### 差距 2：Turn 工作流 —— Plan Steps 未融入对话流

**Codex 模式**：每次请求生成一个「Turn」卡片，内含计划步骤 + 执行状态 + 操作按钮。

**当前状态**：
- `PlanStep` 仅存于独立 `TaskPanel`
- 步骤完成后 1.5s 淡出消失，**对话历史无步骤记录**
- 步骤状态与消息气泡分离，无「Turn」概念

**关键代码**：
```tsx
// TaskPanel.tsx — 步骤完成后即消失
const timer = setTimeout(() => {
  clearPlanSteps();  // 对话历史中无任何留存
  setFadingOut(false);
}, 1500);
```

#### 差距 3：交互式 Diff —— 能看不能操作

**Codex 模式**：每个修改完成后展示 diff 卡片，用户可直接「接受/拒绝/编辑」。

**当前状态**：
- `ToolCallCard` 展示 diff，但 **只有展示无操作**
- Claude 直接通过 CLI 写入文件，用户无「确认后再应用」环节
- `CheckpointPanel` 可还原，但需离开对话上下文

#### 差距 4：统一变更确认 —— 无「Apply All / Undo All」

**Codex 模式**：Turn 完成后出现总览卡片，列出所有修改文件 + diff 统计 + 批量操作按钮。

**当前状态**：
- `ChangeSummaryPanel` 可汇总变更，但为独立面板
- 无「对话流末尾展示变更总览并确认」的环节
- 无批量接受/拒绝概念

#### 差距 5：自动 Checkpoint —— 快照存在但体验割裂

**Codex 模式**：每次 Turn 前自动创建 checkpoint，对话流中直接嵌入回滚入口。

**当前状态**：
- `CheckpointPanel` 为独立面板
- `originalContent` 已捕获，但回滚需离开对话上下文
- 无「Turn 级别快照」概念

## 六、核心结论

> **你已经完成了最难的 80%** —— 将 Claude Code CLI 的全部能力包装进 Electron，并构建了 diff 可视化、步骤追踪、checkpoint 快照等所有「原材料」。
>
> **剩下的 20% 是将这些原材料组装成 Codex 式的对话流体验。**
>
> 这不是「加功能」的问题，而是 **「重构界面范式」** 的问题。当前项目像一只「拥有所有零件但尚未组装的手表」。

## 七、改进路线图

### 方案 A：渐进式改造（推荐，风险低）

**Phase 1：Turn 工作流（2-3 周）**
1. 在 `ChatPanel` 中新增「Turn 卡片」组件（进行中：已具备最小步骤留存和汇总卡片）
2. 将 `PlanStep` 从 `TaskPanel` 迁移到对话流中（进行中：已在 assistant 消息内留存）
3. Turn 完成后在对话末尾追加「步骤总览」卡片（已完成最小版本）
4. 保留现有导航，默认只显示核心项（chat/files/git），其余折叠

**Phase 2：交互式 Diff（1-2 周）**
1. 给 `ToolCallCard` 增加「接受/拒绝/编辑」按钮
2. 实现「接受」时直接应用到文件系统
3. 实现「拒绝」时调用 `CheckpointPanel` 回滚逻辑
4. 将操作结果反馈到对话流

**Phase 3：统一变更确认（1 周）**
1. 每次 Turn 完成后在对话末尾追加「变更总览」卡片
2. 展示修改文件列表 + diff 统计
3. 提供「应用所有变更 / 撤销所有变更」按钮

**Phase 4：体验打磨（1-2 周）**
1. 增加首次启动欢迎向导
2. 导航分组折叠
3. 补充 README 和开发文档
4. 添加核心服务的单元测试

### 方案 B：彻底重构（风险高，目标最纯粹）

1. 将主界面简化为「对话区 + 文件树侧边栏 + 底部输入框」
2. 所有工具调用、diff、变更确认全部内嵌到对话流
3. 其余功能通过设置入口或命令面板访问
4. **适合在 v2.0 大版本时考虑**
