# Claude Code GUI 2.0 路线图

> 目标：围绕“对话即界面”重构交互范式，把现有能力从分散面板重新组装为 Codex 风格的 Turn 工作流。

---

## 一、2.0 目标边界

### 必达目标

1. 每一轮请求都能在对话流中看到完整过程：步骤、工具、Diff、结果、回滚入口。
2. 用户不必在聊天区和独立任务/变更面板之间反复跳转。
3. Claude CLI 原生能力保持不减：工具、MCP、权限审批、多轮上下文继续可用。

### 不属于 2.0 首批阻塞项

1. 自动更新
2. 云同步
3. 插件市场深化
4. 完整设计系统重做

---

## 二、执行原则

### 事实

- 现有项目已经具备工具卡片、Diff 展示、步骤追踪、Checkpoint、权限审批等原材料。
- 真正缺的是“把这些能力按 Turn 重新编排到对话流里”。

### 推导

- 2.0 不是优先继续堆面板，而是优先把核心路径内聚到聊天流。
- 每一阶段都必须能单独交付、单独验证，避免“大重写后再集成”。

---

## 三、阶段路线图

### Phase 0：UI 重构 & 设计系统（✅ 已完成）

目标：重建 UI 交互框架，为 2.0 功能铺设视觉基础。

- [x] 导航收缩 18 项 → 5 项（chat/project/tools/config/history）
- [x] App.tsx 拆分为 NavRail/WorkspaceArea/AuxPanel 三个 layout 组件
- [x] 文件修改工具 Diff 默认展开 + 操作按钮提升至头部
- [x] CSS 设计系统 Token 体系（字号/间距/圆角/动效/阴影）
- [x] NavRail/AuxPanel/WorkspaceArea/ChatPanel 视觉精修
- [x] CSS 语义变量化（7 个主题感知变量，消除硬编码 rgba）
- [x] 亮色主题完整兼容性覆盖
- [x] Vite manualChunks 代码分割（主 bundle 765KB → 283KB）

### Phase 1：Turn 工作流基础

目标：让步骤信息和回合收口留在对话流里。

当前状态：已启动

- [x] assistant 消息支持 `planSteps` 快照留存
- [x] 对话内展示“本轮步骤”区块
- [x] 对话内展示 Turn 结束汇总卡片
- [x] 将步骤留存从"嵌在 assistant 消息内"升级为独立 TurnCard 组件（列表内独立渲染）
- [x] 让 `TaskPanel` 退化为辅助视图（移除 activePlanSteps 实时步骤块，仅保留 todoItems 追踪）

完成标准：

1. 一轮任务结束后，仅看聊天区就能回顾执行过程。
2. 刷新前当前 tab 内历史仍保留步骤轨迹。

### Phase 2：交互式 Diff

目标：让 Diff 从“可看”变成“可操作”。

- [x] 给 `ToolCallCard` 增加接受/拒绝/编辑入口
- [x] 接受后应用修改并回写状态
- [x] 拒绝后基于 Checkpoint 回滚（originalContent 快照机制）
- [x] 操作结果反馈到对话流- [x] `ToolCallView` 文件类工具增加"在编辑器中打开"按钮（fs:openInEditor IPC）
完成标准：

1. 用户不离开聊天区即可处理文件变更。
2. 至少覆盖 Write/Edit/MultiEdit 三类修改工具。

### Phase 3：统一变更确认

目标：让一次 Turn 的所有改动有统一收口。

- [ ] 生成对话内“变更总览”卡片
- [x] 展示文件数、行数增删统计（基于 computeLineDiff 精确计算）
- [ ] 支持“应用全部 / 撤销全部”

完成标准：

1. 用户能对整轮修改做一次性确认。

### Phase 4：极简界面收缩

目标：降低“18 个导航项”带来的认知负荷。

- [ ] 默认只突出聊天、文件、变更三个核心视图
- [ ] 其余能力折叠进二级入口
- [ ] 首次启动增加引导

完成标准：

1. 新用户首次进入不会因导航过多而失焦。

### Phase 5：工程稳态

目标：让 2.0 具备持续重构能力。

- [ ] 拆分 `App.tsx`
- [ ] 为 `CliService` / `FileService` / 核心渲染流补测试
- [ ] README / 开发文档补齐

完成标准：

1. 后续 2.x 迭代不再完全依赖人工回归。
